# Quick start

This guide will get you analyzing code with Vibrant in under 5 minutes.

## Basic usage

<Steps>
  <Step title="Run Vibrant on your codebase">
    The simplest way to use Vibrant is to run it on your current directory:

    ```bash
    npx vibrant-cli .
    ```

    Or if you installed it globally:

    ```bash
    vibrant .
    ```

    <Tip>
      Vibrant automatically detects TypeScript and JavaScript files in common source directories like `src/`, `lib/`, `app/`, etc.
    </Tip>
  </Step>

  <Step title="Review the results">
    Vibrant will analyze your code and display any issues found:

    ```bash
    🔮 Vibrant Analysis
    ───────────────────

    ✔ Analysis complete

    ✕ src/api.ts:24:5
      └─ hardcoded-credentials
         API key hardcoded in source code

         24|   const API_KEY = "sk-abc123...";

         → Use environment variables instead

    ⚠ src/utils.ts:85:1
      └─ empty-catch-block
         Empty catch block swallows errors

         85|   } catch (e) {}

         → Log or handle the error properly

    ✕ 1 error · ⚠ 1 warning · 48 files · 200ms
    ```

    <Note>
      - ✕ (red) indicates **errors** - critical issues that should be fixed
      - ⚠ (yellow) indicates **warnings** - issues that should be reviewed
    </Note>
  </Step>

  <Step title="Fix issues automatically (optional)">
    Some issues can be automatically fixed:

    ```bash
    vibrant . --fix
    ```

    This will fix issues like:
    - Removing `console.log` statements
    - Adding error handling to empty catch blocks
    - Other auto-fixable patterns

    ```bash
    ✓ fixed 5 issues
    ```
  </Step>
</Steps>

## Analyze specific files or directories

You can target specific paths:

<CodeGroup>
```bash Single file
vibrant src/api.ts
```

```bash Specific directory
vibrant src/components
```

```bash Glob pattern
vibrant "src/**/*.ts"
```

```bash Multiple paths
vibrant src/api.ts src/utils.ts
```
</CodeGroup>

## Ignore files

Exclude certain files or directories from analysis:

```bash
vibrant . --ignore "dist,coverage,*.test.ts"
```

Or configure it in `vibrant.config.js`:

```javascript vibrant.config.js
module.exports = {
  ignore: ['node_modules', '.git', 'dist', '*.test.ts', 'coverage']
};
```

## AI-powered analysis

<Warning>
  AI analysis requires an API key. See the [Installation](/installation#ai-provider-setup-optional) guide for setup instructions.
</Warning>

Enable AI-powered deep analysis for more sophisticated pattern detection:

<Steps>
  <Step title="Set up your API key">
    Create a `.env` file in your project:

    ```bash .env
    OPENROUTER_API_KEY="sk-or-v1-..."
    ```

    <Tip>
      OpenRouter offers free models and is recommended for getting started. See [Installation](/installation#get-api-keys) for other providers.
    </Tip>
  </Step>

  <Step title="Run AI analysis">
    ```bash
    vibrant . --ai
    ```

    You'll see enhanced analysis with:
    - Overall code health assessment
    - Key findings and patterns
    - Actionable recommendations
  </Step>

  <Step title="Review AI insights">
    ```bash
    🔮 Vibrant Analysis
    ───────────────────

    ✔ Analysis complete

    🔍 Issues Found

    [... issues ...]

    📊 Diagnosis

    The codebase shows signs of AI-generated code with
    multiple hardcoded credentials and incomplete error
    handling patterns.

    ⚡ Key Findings

    • 3 hardcoded API keys need to be moved to environment variables
    • 5 empty catch blocks silently swallow errors
    • 12 console.log statements should be removed

    💡 Recommendations

    → Move all credentials to environment variables using dotenv
    → Add proper error logging in catch blocks
    → Remove debug console.log statements before production

    ✕ 8 errors · ⚠ 12 warnings · 48 files · 10.2s
    ```
  </Step>
</Steps>

## Output formats

Vibrant supports different output formats for different use cases:

<Tabs>
  <Tab title="Pretty (default)">
    Human-readable output with colors:

    ```bash
    vibrant .
    ```

    Best for: Interactive terminal usage
  </Tab>

  <Tab title="Compact">
    Single-line output for each issue:

    ```bash
    vibrant . --format compact
    ```

    Output:
    ```
    src/api.ts:24:5 error hardcoded-credentials API key in source
    src/utils.ts:85:1 warning empty-catch-block Swallows errors
    ```

    Best for: CI/CD pipelines, grep-able output
  </Tab>

  <Tab title="JSON">
    Machine-readable JSON output:

    ```bash
    vibrant . --format json > report.json
    ```

    Best for: Tooling integration, custom reporting
  </Tab>

  <Tab title="Plan">
    Detailed markdown report:

    ```bash
    vibrant . --format plan
    ```

    Creates `vibrant-report.md` with:
    - Complete issue list with code snippets
    - Suggested fixes for each issue
    - Priority ordering (errors first)

    Best for: AI assistants to auto-fix issues, documentation
  </Tab>
</Tabs>

## Ignoring false positives

Sometimes you need to ignore specific lines:

<CodeGroup>
```typescript Ignore current line
// vibrant ignore
const debug = true;
```

```typescript Ignore next line
// vibrant ignore-next-line
const apiKey = process.env.API_KEY || "default-key";
```

```typescript Inline ignore
const temp = "test"; // vibrant ignore
```
</CodeGroup>

## CI/CD integration

Integrate Vibrant into your continuous integration pipeline:

<CodeGroup>
```yaml GitHub Actions
name: Code Quality

on: [push, pull_request]

jobs:
  analyze:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Run Vibrant
        run: npx vibrant-cli . --format compact
```

```yaml GitLab CI
code-quality:
  image: node:20
  script:
    - npx vibrant-cli . --format compact
  only:
    - merge_requests
    - main
```
</CodeGroup>

<Note>
  Vibrant exits with code `1` if errors are found, and `0` if no issues or only warnings. This integrates naturally with CI/CD systems.
</Note>
